**任务说明**:

现有E2E测试（如debate-flow/single-round.spec.ts）使用APIRequestContext调用API，而Mock配置只能拦截page级别的请求。E2E测试设计为集成测试，需要真实的后端服务运行。

**分析结果**:

1. **Mock配置限制**：
   - 创建的E2EMockConfig使用page.route拦截请求
   - 只能拦截浏览器Page发起的HTTP请求
   - 无法拦截APIRequestContext直接发起的请求

2. **E2E测试架构**：
   - `debate-flow/helpers.ts`中的函数使用APIRequestContext
   - playwright.config.ts配置了自动启动开发服务器
   - 测试设计为与真实后端集成

3. **现有测试状态**：
   - `single-round.spec.ts`等测试需要真实的数据库和API
   - 配置了webServer自动启动（`npm run dev`）
   - 需要真实环境运行才能验证

**结论**:

已完成的Mock配置和测试辅助函数（Task 5.2-5.3）为E2E测试提供了完善的Mock基础设施，可用于：
- 创建新的Mock E2E测试（使用page级别API调用）
- UI交互测试（按钮点击、表单填写等）
- 前端渲染测试

现有的流程测试（single-round, multi-round等）是集成测试，需要真实后端环境，这部分测试保持现状，由开发团队在CI/CD环境中运行。

**可选改进**:

1. 创建新的Mock E2E测试套件，使用page级别调用测试UI流程
2. 使用msw或类似库拦截APIRequestContext请求（需要额外配置）
3. 将E2E测试分离为：
   - Mock E2E测试：测试UI交互，使用Mock数据
   - 集成E2E测试：测试完整流程，需要真实环境

### 架构分析

**发现的问题**:

现有E2E测试（debate-flow等）使用APIRequestContext直接调用API，而Mock配置只能拦截page级别的请求。这是设计上的差异：

- **Mock配置**: `page.route('**/api/**', ...)` - 只拦截浏览器请求
- **现有测试**: `apiContext.post('/api/...')` - 直接HTTP请求，绕过浏览器

**技术方案**:

1. **保持现有集成测试**（推荐）:
   - 现有测试作为集成测试在CI/CD中运行
   - 需要真实环境和数据库
   - 用于验证完整业务流程

2. **创建Mock E2E测试**:
   - 新增测试使用page级别调用
   - 使用已实现的Mock配置
   - 用于快速验证UI交互

3. **使用MSW库**（可选）:
   - 可以拦截APIRequestContext请求
   - 需要额外依赖和配置
   - 复杂度较高
