# API 版本策略指南

> 制定日期：2026-02-21
> 状态：**当前执行规范**

---

## 一、核心规则（一句话版本）

> **新功能永远写在 `/api/v1/`，不要在 `/api/` 根级新增任何路由。**

---

## 二、三层路由结构

```
/api/auth/          ← 认证层（登录、注册、token刷新）保持现状，不纳入版本管理
/api/v1/            ← 业务功能层（唯一正式版本，所有前端应调用此层）
/api/admin/         ← 管理员专属（数据统计、用户管理，不对普通用户暴露）
/api/               ← 遗留层（已有路由保留兼容，禁止新增）
```

---

## 四、开发者行动指南

### ✅ 新增功能时

```
src/app/api/v1/my-feature/route.ts   ← 在这里写实现
```

### ✅ 修改现有功能时（非破坏性）

直接修改对应的根级路由文件，v1 会自动获得更新（因为是转发关系）。

### ✅ 对现有功能做破坏性升级时（改变接口格式/参数）

```
1. 在 /api/v1/xxx/route.ts 写新实现（此时不再是 export * 转发）
2. 根级路由 /api/xxx/route.ts 保持旧实现，不改动
3. 前端逐步迁移到 v1 的新接口
4. 待全部迁移完成后，根级路由可标记废弃
```

### ❌ 禁止行为

- 在 `/api/` 根级新增 `route.ts` 文件
- 在 `/api/v1/` 和 `/api/` 同时维护两套实现（除破坏性升级场景外）
- 将业务逻辑路由直接放在 `/api/admin/`（admin 只用于管理操作）

---

## 五、未来升级路径（v2）

当 v1 某个模块需要做重大重构时：

```
/api/v1/cases/   ← 继续服务旧前端，不动
/api/v2/cases/   ← 新实现，新前端使用
```

v1 和 v2 可并行运行，前端按模块逐步迁移，不需要整体重写。

## 六、当前版本状态

| 版本          | 状态        | 说明                         |
| ------------- | ----------- | ---------------------------- |
| `/api/`       | 🟡 遗留     | 保留向后兼容，禁止新增       |
