# 管理后台治理增强清单

> 目标：在“后台可用”的基础上，进一步把管理后台提升到“正式部署可治理”的水平。
>
> 说明：
>
> - `P0`：建议正式上线前完成
> - `P1`：建议上线后尽快补齐
> - `P2`：中长期治理增强项

---

## P0

### 1. 敏感操作二次确认

**问题**

- 当前高风险操作大多仍是一次点击即生效
- 误操作、内鬼风险、越权后破坏面偏大

**范围**

- 修改管理员 / 超级管理员角色
- 删除管理员账号
- 修改系统核心配置
- 修改系统内置角色及其权限
- 批量角色调整 / 批量会员状态调整 / 批量订单状态调整

**完成标准**

- 高风险操作统一弹出二次确认
- 需要填写操作原因
- 支持强制输入目标对象标识二次确认
- 服务端审计日志保留 `before / after / operator / reason`

### 2. 管理员强认证

**问题**

- 目前后台主要依赖登录态和权限校验
- 缺少高危操作再验身份

**完成标准**

- 管理后台登录支持 MFA
- 高危操作支持二次密码验证或一次性验证码
- 后台会话支持设备列表、最近登录记录、强制下线
- 异地 / 异常设备登录触发告警

### 3. 配置变更版本化与回滚

**问题**

- 系统配置能改，但治理能力不足
- 缺少版本、回滚、发布记录

**完成标准**

- 每次配置变更生成历史版本
- 记录修改前值、修改后值、修改人、修改原因、修改时间
- 支持按 key 查看历史
- 支持一键回滚到历史版本
- 回滚动作同样写入审计日志

### 4. 运维策略持久化

**问题**

- `rate-limit`、`IP filter`、`reputation` 这类后台控制项如果主要依赖内存态，会有重启丢失和多实例不一致风险

**完成标准**

- 速率限制配置持久化到数据库或统一配置中心
- IP 黑白名单持久化
- 用户信誉/风控策略持久化
- 支持多实例读取一致状态
- 支持后台查看当前生效值与最后修改人

### 5. 审计日志完整化

**问题**

- 现在已有日志，但高危后台动作仍不够细

**完成标准**

- 所有后台高危操作必须写审计日志
- 审计日志包含：
  - 操作者
  - 目标对象
  - 操作类型
  - 原因
  - 变更前后差异
  - 请求来源 IP / UA
- 审计日志不可被普通管理员删除或篡改

## P1

### 6. 后台角色细分

**问题**

- 当前后台仍偏 `ADMIN / SUPER_ADMIN` 两层模型
- 权限粒度不够细

**建议角色**

- 只读审计员
- 用户运营管理员
- 订单 / 财务管理员
- 配置管理员
- 告警处理员
- 内容 / 法条审核员

**完成标准**

- 后台能力按角色域拆分
- 低风险后台查看能力可授予只读角色
- 高危能力默认仅 `SUPER_ADMIN`

### 7. 字段级脱敏

**问题**

- 后台天然能看到手机号、邮箱、地址、IP、请求参数等敏感字段

**完成标准**

- 用户列表默认脱敏展示手机号、邮箱、地址
- 日志页面默认脱敏 token、cookie、身份证号、银行卡号、手机号
- 仅特定权限可查看明文
- 明文查看动作写审计日志

### 8. 告警闭环治理

**问题**

- 当前更偏“看告警、确认、处理”，缺少完整流程

**完成标准**

- 告警支持负责人
- 支持 SLA / 超时升级
- 支持误报标记
- 支持关联错误日志、关联配置变更、关联发布时间点
- 支持告警抑制与重复合并

### 9. 批量操作安全阈值

**问题**

- 批量接口虽然有权限，但还缺统一防误操作机制

**完成标准**

- 批量操作有数量阈值
- 超过阈值需额外确认
- 批量操作支持 dry-run 预览
- 服务端返回影响对象清单摘要

## P2

### 10. 后台审批工作流

**目标**

- 将“直接执行型后台”升级为“申请 -> 审批 -> 生效”

**适用操作**

- 超级管理员授权
- 系统配置核心项修改
- 批量封禁 / 批量恢复
- 导出高敏感数据

### 11. 后台健康总览

**目标**

- 让后台不只管业务，也能管系统健康

**建议内容**

- 作业队列
- AI 调用成功率
- 文档分析失败率
- 提炼 / 风险评估失败率
- 导出任务积压
- 告警趋势

### 12. 后台操作回放与追责

**目标**

- 出问题时能回答“谁在什么时候改了什么”

**完成标准**

- 审计日志支持 diff 展示
- 支持按管理员时间线查看
- 支持导出单次事件链

---

## 建议排期

### 第一批（上线前）

1. 敏感操作二次确认
2. 管理员强认证
3. 配置变更版本化与回滚
4. 运维策略持久化
5. 审计日志完整化

### 第二批（上线后尽快）

1. 后台角色细分
2. 字段级脱敏
3. 告警闭环治理
4. 批量操作安全阈值

### 第三批（平台化治理）

1. 后台审批工作流
2. 后台健康总览
3. 后台操作回放与追责

---

## 当前建议

如果只选一个最值得立刻做的组合，我建议是：

1. `管理员强认证`
2. `配置变更版本化与回滚`
3. `运维策略持久化`

这是最能降低正式上线风险的一组。
