# 真实卷宗工作流完善 — 原子化实施路线图

> 文档用途：交给 Claude / 工程同事直接实施  
> 目标：让“律师上传一整套卷宗材料后，系统能自然推进到案情提炼、争议焦点、风险评估、辩论推演、整案交付包”  
> 编写时间：2026-05-12  
> 基于最近联调结论整理

---

## 一、背景结论

当前系统已经通过联调验证了以下能力：

- 律师账号可登录并创建案件
- 可批量上传卷宗材料
- 文档分析最终可进入 `COMPLETED`
- 可创建辩论并生成论点
- 可生成整案交付包
- 可提交复核、导出 DOCX、内容变更后触发“需重新复核”

但在“更像真实律师办案”的完整工作流中，仍存在三个明显短板：

1. 上传卷宗后，系统不会自动把材料沉淀为完整案件提炼结果
2. 辩论生成质量偏模板化，未充分抓住卷宗中的核心争议
3. 风险评估没有自然进入案件主流程，交付包 `§5` 长期为空

本路线图的目标，就是补齐这三条主链。

---

## 二、目标定义

### 2.1 用户目标

一个非技术律师完成以下动作后：

1. 新建案件
2. 上传 5-10 份卷宗材料

系统应能在无需开发介入的前提下，逐步生成：

- 案件摘要
- 已确认事实 / 存疑事实
- 争议焦点
- 风险评估
- 初始辩论
- 更完整的整案交付包

### 2.2 30 天内的完成标准

- 上传卷宗后，案件页可见“案件提炼中 / 已完成”
- 案件提炼完成后，交付包 `§2` 不再长期 fallback
- 风险评估结果可进入案件主流程，交付包 `§5` 不再长期为空
- 辩论生成不再出现模板变量残留如 `${sideName}`
- 使用 `test-data/realistic-dossiers/loan-dispute-case-001/` 测试时，交付包完整度从 `5/7` 提升到 `7/7`

## 三、实施原则

1. 不增加新的“孤立功能页”，优先补齐现有案件工作流
2. 所有推进动作都以 `Case` 为中心，不依赖人工记住额外入口
3. 尽量复用已有：
   - `Document.analysisResult`
   - `Conversation.caseContext`
   - `Debate.summary`
   - `Case.metadata`
4. 所有新增能力必须可被整案交付包消费
5. 所有阶段都要能被 `test-data/realistic-dossiers/loan-dispute-case-001/` 验证

### 3.1 追加实施约束（必须遵守）

#### A. 数据落点统一

本期案件提炼结果统一写入：

- `Case.metadata.extractionSnapshot`

禁止在本期同时新增第二套并行持久化结构（例如既写 `Case.metadata` 又写独立快照表），避免后续 `package-builder`、风险评估和案件页读取来源分裂。

#### B. 自动触发策略

文档上传成功响应不得依赖以下任务执行完成：

- 案件提炼
- 风险评估

它们必须以**异步 fire-and-forget** 方式触发：

- 成功：更新案件级状态 / 结果快照
- 失败：只记录日志和状态
- 不得改变上传接口成功返回

#### C. 案件提炼触发阈值

自动触发案件提炼的最小条件：

1. 同一案件下 `analysisStatus = COMPLETED` 的文档数不少于 `3`

或

2. 已存在两类以上核心材料：
   - 案件概览
   - 起诉状 / 答辩状
   - 协议 / 借条
   - 银行流水
   - 聊天记录

禁止在仅上传 1 份零散材料时就自动触发完整案件提炼。

#### D. 辩论修复优先级

Phase 3 的实施顺序必须固定为：

1. 先修**功能正确性**
   - 模板变量残留
   - 输入源接入不完整
2. 再做**质量优化**
   - 被告抗辩识别
   - 自动摘要联动

禁止在模板错误尚未清零时直接进入复杂质量优化。

#### E. 最低可交付标准

若 30 天内无法稳定达到交付包完整度 `7/7`，则最低可交付标准为：

- `§1` 必须为 `primary`
- `§4` 必须为 `primary`
- `§5` 必须为 `primary`
- `§2 / §3 / §6` 不低于 `enhanced`
- `§7` 必须稳定输出

并在回归验收中明确记录：

- 哪些章节尚未达到 `primary`
- 原因是什么
- 是否阻塞真实试点

| 阶段    | 名称               | 目标                                          |

## 六、Phase 1 — 案件提炼工作流

### 目标

上传卷宗材料后，系统自动生成案件级提炼结果，并使交付包 `§2` 获得稳定数据来源。

### 设计原则

- 以 `Case` 为中心推进
- 不要求用户进入 chat 才能产生案情晶体
- 优先复用已完成的 `Document.analysisResult`

### 原子任务

#### 1.1 新增案件级提炼服务

新建服务层：

- `src/lib/case/case-extraction-service.ts`

职责：

- 聚合案件下所有 `analysisStatus = COMPLETED` 的文档
- 提取：
  - parties
  - claims
  - summary
  - keyFacts
  - dispute clues
- 输出统一的 `CaseExtractionSnapshot`

**输出建议结构**

```ts
{
  summary: string | null,
  plaintiffName: string | null,
  defendantName: string | null,
  establishedFacts: string[],
  uncertainFacts: string[],
  disputeFocuses: string[],
  sourceDocumentIds: string[],
  generatedAt: string
}
```

**验收标准**

- 给定同一案件多份文档，能稳定输出单一聚合结果

#### 1.2 新增案件级提炼 API

新建：

- `POST /api/v1/cases/[id]/extract`

建议路径：

- `src/app/api/v1/cases/[id]/extract/route.ts`

职责：

1. 鉴权
2. 校验案件访问权限
3. 调用 `case-extraction-service`
4. 将提炼结果落到 `Case.metadata` 或专用快照字段
5. 返回聚合结果

**验收标准**

- 调用后返回结构化提炼结果
- 案件记录能持久化保存

#### 1.3 上传文档完成后触发案件提炼

在文档上传链路中，补一个“异步后置触发”：

- 文档分析完成后，不只写回 `Document.analysisResult`
- 同时触发案件级提炼

**涉及文件**

- `src/app/api/v1/documents/upload/route.ts`

**注意**

- 失败不应阻断上传主流程
- 需要记录日志，便于排障

**验收标准**

- 上传 3 份以上材料后，案件不需要额外人工点击即可具备提炼结果

#### 1.4 让交付包 `§2` 优先读案件级提炼结果

当前 `§2` 主要依赖：

- `Debate.summary.keyLegalIssues`
- `CaseCrystal.core_dispute`

调整为：

优先级：

1. `Case.metadata.extractionSnapshot.disputeFocuses`
2. `Debate.summary.keyLegalIssues`
3. `CaseCrystal.core_dispute`
4. fallback

**涉及文件**

- `src/lib/case/package-builder.ts`

**验收标准**

- 没有辩论时，`§2` 仍可由卷宗提炼结果支撑，不再长期 fallback

## 七、Phase 2 — 风险评估接入主流程

### 目标

卷宗上传并完成案件提炼后，系统能自然生成案件风险评估，并稳定写入案件。

### 原子任务

#### 2.1 新增案件风险评估服务封装

新建：

- `src/lib/case/case-risk-pipeline.ts`

职责：

- 读取案件提炼结果
- 转成风险评估 API 所需输入
- 统一返回案件级风险快照

**验收标准**

- 不需要前端手写 `facts / claims / evidence`，服务端可自动组装

#### 2.2 新增案件级风险评估 API

新建：

- `POST /api/v1/cases/[id]/assess`

职责：

1. 鉴权
2. 校验案件访问权限
3. 读取案件提炼结果
4. 调用风险评估服务
5. 将结果写回 `Case.metadata.riskAssessment`

**验收标准**

- 返回 `winRate / difficulty / riskLevel / keyRisks`

#### 2.3 上传卷宗完成后自动触发风险评估

策略：

- 在案件提炼成功后自动触发
- 或在文档数量 ≥ N（建议 3）后触发

**涉及文件**

- `src/app/api/v1/documents/upload/route.ts`
- 或 `src/lib/case/case-extraction-service.ts`

**验收标准**

- 新案件上传主材料后，`§5` 不再长期为空

#### 2.4 统一交付包 `§5` 数据结构

当前 `§5` 读取逻辑过于依赖静态 metadata 结构。  
需要统一为固定 schema，例如：

```ts
{
  winRate: number,
  difficulty: string,
  riskLevel: string,
  aiAssessment: {
    summary: string,
    keyRisks: string[],
    generatedAt: string
  }
}
```

**涉及文件**

- `src/lib/case/package-builder.ts`

**验收标准**

- 卷宗案件中 `§5` 可稳定读取，不依赖人工额外补 metadata

## 八、Phase 3 — 辩论生成质量修复

### 目标

让辩论模块不只是“能出论点”，而是能真正围绕卷宗里的关键争议生成可用内容。

### 当前联调暴露的问题

- 出现模板变量残留：`${sideName}`
- 论点过于泛化
- 没有抓住：
  - 被告自认剩余本金
  - 现金还款 20,000 元无证据
  - 利率过高仅影响利息、不影响本金成立

### 原子任务

#### 3.1 修复模板变量未替换问题

检查并修复：

- `src/app/api/v1/debate-rounds/[roundId]/generate/route.ts`

重点检查：

- 默认 fallback 文案里是否仍保留 `${sideName}` 等占位符

**验收标准**

- 任意自动生成论点中不出现模板变量残留

#### 3.2 引入案件提炼结果作为论点生成输入

当前论点生成主要依赖：

- claims
- parties
- keyFacts
- articles

需要补充读取：

- disputeFocuses
- establishedFacts
- uncertainFacts
- risk summary

**验收标准**

- 生成论点时能显式围绕争议点，而不是泛泛而谈

#### 3.3 加强“被告抗辩”识别

在 `06_被告答辩意见_模拟版.txt` 场景下，系统应识别：

- 已认可借款关系
- 争的是现金还款 20,000 元
- 争的是利率过高

**验收标准**

- 被告方论点至少有 2 条直接回应这三个点

#### 3.4 辩论结束后自动生成摘要

创建完首轮辩论后，自动调用：

- `POST /api/v1/debates/[id]/summary`

确保：

- `Debate.summary` 可被后续交付包直接消费

**验收标准**

- 新建辩论并生成一轮后，`§6` 从 fallback 升为 primary

## 九、Phase 4 — 案件页进度面板

### 目标

让非技术律师一打开案件页就知道“下一步要做什么”。

### 原子任务

#### 4.1 新增案件流程状态聚合器

新建：

- `src/lib/case/case-workflow-status.ts`

聚合字段：

- 文档上传数
- 文档分析完成数
- 是否有案件提炼
- 是否有风险评估
- 是否有辩论
- 是否有交付包复核

#### 4.2 在案件详情页新增“办案进度”面板

涉及页面：

- `src/app/cases/[id]/page.tsx`

展示：

- 步骤 1：上传卷宗
- 步骤 2：案件提炼
- 步骤 3：风险评估
- 步骤 4：辩论推演
- 步骤 5：整案交付包

每一步显示：

- 未开始 / 进行中 / 已完成
- 下一步按钮

**验收标准**

- 非技术用户无需猜测下一步入口

#### 4.3 提供一键推进入口

例如：

- “生成案件摘要”
- “生成风险评估”
- “创建初始辩论”
- “打开整案交付包”

**验收标准**

- 每个关键动作都可从案件页进入

1. 先完成 `Phase 1`
