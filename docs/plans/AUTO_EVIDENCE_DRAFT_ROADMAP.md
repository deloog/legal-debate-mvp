# 自动证据草稿（Document -> Evidence）原子化实施路线图

## 背景

当前“真实卷宗自动化”链路已经可稳定达到：

- 文档上传
- 文档分析
- 案件提炼
- 风险评估
- 辩论推演
- 整案交付包 `6/7`

唯一稳定缺口是：

- `§4 证据清单` 仍依赖律师手动新增 `Evidence`

这不是数据完全缺失，而是“文档分析结果中的证据线索”还没有自动沉淀到 `Evidence` 表。

当前代码里已存在可复用能力：

- `Document.analysisResult.extractedData.claims[].evidence`
- `Document.analysisResult.extractedData.disputeFocuses[].evidence`
- `Document.analysisResult.extractedData.keyFacts[].evidence`
- `Evidence` 表已有：
  - `type`
  - `name`
  - `description`
  - `status`
  - `relevanceScore`
  - `metadata`
- `package-builder` 的 `§4` 已直接消费 `Evidence` 表

因此本期目标不是“重做证据系统”，而是：

**在文档分析完成后，自动生成“待律师确认的证据草稿”，让 `§4` 在真实卷宗场景下可自动变为 available。**

---

## 目标

### 主目标

让真实卷宗工作流从：

`6/7（§4 缺失）`

提升到：

`7/7（§4 自动可用）`

### 次目标

- 自动生成的证据必须是“草稿”，不能直接假装已被律师确认
- 自动证据应尽量去重，避免一个案件生成大量重复证据
- 现有 `EvidenceTab`、交付包 `§4`、证据接口无需大改即可消费

---

## 范围边界

### 本期要做

- 从 `analysisResult` 中自动抽取证据候选
- 生成 `Evidence` 草稿记录
- 用最小规则做去重
- 接入文档分析完成后的异步链路
- 在证据页区分“自动草稿”和“人工证据”
- 让交付包 `§4` 自动吃到这些草稿证据

### 本期不做

- 不做复杂证据链图谱自动构建
- 不做证据真实性自动裁判
- 不做跨案件证据合并
- 不自动把草稿状态升级为已采纳
- 不重构现有 Evidence API 为 v2

---

## 追加实施约束（必须遵守）

### A. 自动生成的是“草稿”，不是正式证据结论

本期自动创建的 `Evidence` 统一使用：

- `status = PENDING`
- `metadata.autoGenerated = true`

禁止自动写成：

- `ACCEPTED`
- `QUESTIONED`
- `REJECTED`

### B. 仅复用现有 Evidence 表，不新增第二套平行证据表

本期统一写入现有 `Evidence` 表。  
扩展信息全部放入 `metadata`，禁止新增 `AutoEvidenceDraft` 等平行模型。

### C. 必须异步触发，不阻断文档上传/分析主流程

文档分析完成后的自动证据生成必须以 fire-and-forget 方式触发。  
失败仅记录日志与状态，不得影响：

- 文档上传成功响应
- 文档分析完成状态
- 提炼 / 风险评估已有链路

### D. 自动证据必须支持去重

同一案件内，若多个文档分析结果都指向同一证据项，不得盲目重复创建。  
至少要按“规范化证据名”做一次去重。

### E. 证据草稿优先采用结构化线索，不依赖纯 OCR 文本猜测

优先使用：

1. `claims[].evidence`
2. `disputeFocuses[].evidence`
3. `keyFacts[].evidence`

仅当这些都缺失时，才允许使用保守的文件名级 fallback。

## 数据设计

### Evidence.metadata 新增约定（不改 schema）

自动生成证据时，`metadata` 统一包含：

```json
{
  "autoGenerated": true,
  "sourceDocumentIds": ["doc_xxx"],
  "sourceFilenames": ["03_银行转账流水_整理版.txt"],
  "sourceKinds": ["claim", "keyFact", "disputeFocus"],
  "evidenceHints": ["银行转账凭证"],
  "supportsFacts": ["2023年1月借款10万元"],
  "supportsDisputes": ["借款是否已还清"],
  "confidence": 0.82,
  "normalizedName": "银行转账流水",
  "draftVersion": "v1"
}
```

### 自动证据名称规范

本期采用“规则优先”的规范化命名：

- 命中 `借条|借款协议|协议` → `借条与借款协议`
- 命中 `银行|转账|流水|回单` → `银行转账流水`
- 命中 `微信|聊天|短信|记录` → `聊天记录`
- 命中 `起诉状` → `起诉状`
- 命中 `答辩状|答辩意见` → `答辩意见`
- 命中 `证据目录` → `证据目录`
- 否则回退到原始线索文本或文件名简化结果

### Evidence.type 映射规则

- 文字材料 / 合同 / 借条 / 起诉状 / 答辩状 / 证据目录 / 流水 → `DOCUMENT`
- 聊天记录 / 电子截图 / 微信记录 → `AUDIO_VIDEO` 或 `DOCUMENT` 均可

本期建议统一保守写成：

- `DOCUMENT`

理由：

- 能与现有 `EvidenceTab` 和交付包兼容
- 减少分类误判

后续再做更细类型优化。

  - `claims[].evidence`
  - `disputeFocuses[].evidence`

目标：

- 确认 `EvidenceTab` 不会因为 `metadata.autoGenerated` 等新字段报错

通过标准：

- 现有 `Evidence` 列表渲染仅读取通用字段，额外 metadata 不影响页面

## Phase 1 — 自动证据提取服务

### 1.1 新建 `src/lib/evidence/auto-evidence-draft-service.ts`

职责：

- 从一个案件下所有 `COMPLETED` 文档中提取证据候选
- 汇总多来源证据线索
- 规范化名称
- 去重
- 输出 `EvidenceDraft[]`

建议输出类型：

```ts
type EvidenceDraft = {
  normalizedName: string;
  type: 'DOCUMENT';
  description: string | null;
  sourceDocumentIds: string[];
  sourceFilenames: string[];
  sourceKinds: Array<'claim' | 'disputeFocus' | 'keyFact' | 'filename'>;
  evidenceHints: string[];
  supportsFacts: string[];
  supportsDisputes: string[];
  confidence: number;
};
```

### 1.2 实现证据线索提取函数

建议拆分纯函数：

- `extractEvidenceHintsFromClaim()`
- `extractEvidenceHintsFromDisputeFocus()`
- `extractEvidenceHintsFromKeyFact()`
- `normalizeEvidenceName()`
- `mergeEvidenceDrafts()`

### 1.3 仅使用保守规则估算 `confidence`

建议：

- 同时来自 `claim + keyFact`：`0.85`
- 来自任意两个不同来源：`0.8`
- 仅来自一个结构化来源：`0.72`
- 仅来自文件名 fallback：`0.55`

本期不要做复杂 AI 二次评分。

### 1.4 实现 `generateAutoEvidenceDrafts(caseId)`

目标：

- 查询案件文档
- 提取候选
- 去重合并
- 返回草稿数组

验收标准：

- 对 `loan-dispute-case-001` 这套卷宗，至少能稳定产出：
  - 借条与借款协议
  - 银行转账流水
  - 聊天记录
  - 起诉状 / 答辩意见 / 证据目录 中的合理子集

## Phase 2 — 落库与幂等

### 2.1 实现 `upsertAutoEvidenceDrafts(caseId)`

职责：

- 调用 `generateAutoEvidenceDrafts(caseId)`
- 与现有 `Evidence` 记录比对
- 若同案已存在相同 `normalizedName` 的自动草稿，则更新 metadata 聚合来源
- 若不存在，则创建新 `Evidence`

### 2.2 去重规则

本期最小规则：

- 同一 `caseId`
- `deletedAt = null`
- `metadata.autoGenerated = true`
- `metadata.normalizedName` 相同

则视为同一草稿证据

### 2.3 创建记录字段约定

写入 Evidence 时建议：

- `type = DOCUMENT`
- `name = normalizedName`
- `description = 自动从卷宗材料中识别，待律师确认`
- `status = PENDING`
- `relevanceScore = confidence`
- `source = 自动识别`
- `submitter = AI系统`
- `metadata = {...}`

### 2.4 人工证据优先原则

若同名证据已存在且不是自动生成的：

- 不覆盖人工证据
- 自动草稿应跳过或仅记录日志

避免 AI 把人工录入的正式证据污染掉。

验收标准：

- 重复触发不会无限新增重复草稿
- 手工新增的证据不会被自动草稿覆盖

---

## Phase 3 — 接入文档分析链路

### 3.1 在文档分析完成后异步触发自动证据生成

接入点建议：

- `src/app/api/v1/documents/upload/route.ts`
- 在文档分析写回 `COMPLETED` 后
- 提炼 / 风险评估链路之后或并行 fire-and-forget 触发

建议链路：

`document analysis completed`
-> `runCaseExtraction`
-> `runCaseRiskAssessment`
-> `upsertAutoEvidenceDrafts`

或

`document analysis completed`
-> 并行 fire-and-forget：

- extraction
- risk
- autoEvidence

本期优先不改动提炼/风险既有成功语义。

### 3.2 增加最小触发阈值

自动证据生成不要求等全部文档完成。  
建议阈值：

- 同案 `COMPLETED` 文档 >= 2

理由：

- 借条 + 起诉状
- 流水 + 聊天记录

都已经足以生成第一批草稿证据

### 3.3 错误处理

要求：

- 任一失败仅 `logger.warn/error`
- 不回滚文档分析结果
- 不阻断提炼/风险评估

---

## Phase 4 — 前端证据页最小产品化

### 4.1 在 EvidenceTab 中区分自动草稿与人工证据

建议：

- `metadata.autoGenerated = true` 的记录显示“自动草稿”徽标
- `status = PENDING` 时显示“待确认”

### 4.2 增加最小提示文案

例如：

- “以下证据由系统从卷宗材料中自动识别，建议律师复核后再作为正式证据使用”

### 4.3 可选：增加一个快速确认入口

本期可只做展示，不强制做复杂审核流。  
如果实现成本低，可加：

- “标记为已确认”

但这不是本期阻塞项。

验收标准：

- 律师打开证据页时，能明显分辨哪些是系统自动识别的草稿
