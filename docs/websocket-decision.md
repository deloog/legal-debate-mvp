# WebSocket 技术决策文档

## 决策结果：当前阶段不引入 WebSocket

**决策日期**: 2024-01-27
**决策人**: 技术架构评估
**项目阶段**: MVP

---

## 📊 决策依据

### 1. 项目特点分析

| 维度 | 评估 |
|------|------|
| **项目阶段** | MVP（最小可行产品） |
| **核心价值** | AI 法律辩论生成 + 案件管理 |
| **用户场景** | 专业法律工作者，重质不重速 |
| **交互特点** | 异步协作为主，非即时通讯 |

### 2. 实时性需求评估

```
高实时性需求（< 1秒）:
  ❌ 无此类场景

中实时性需求（1-30秒）:
  ✅ 辩论生成（已用 SSE 解决）
  ✅ 案件讨论（可接受延迟）

低实时性需求（> 30秒）:
  ✅ 通知提醒
  ✅ 团队协作状态
```

### 3. 技术成本对比

| 方案 | 开发 | 维护 | 资源 | 复杂度 | 总评 |
|------|------|------|------|--------|------|
| **SSE + 轮询** | 低 | 低 | 低 | 低 | ⭐⭐⭐⭐⭐ |
| **WebSocket** | 中 | 中 | 高 | 高 | ⭐⭐⭐ |

---

## 🎯 采用方案：SSE + 智能轮询

### 方案架构

```
┌─────────────────────────────────────────────┐
│                客户端                        │
├─────────────────────────────────────────────┤
│  1. SSE 流式推送                             │
│     用途：辩论生成实时推送                   │
│     端点：/api/v1/debates/[id]/stream       │
│                                              │
│  2. 智能轮询                                 │
│     用途：讨论、通知、团队状态更新          │
│     机制：                                   │
│       - 页面激活：每 30s 轮询               │
│       - 页面隐藏：停止轮询                  │
│       - 用户操作后：立即触发                │
│                                              │
│  3. 手动刷新                                 │
│     备选方案：用户主动刷新按钮              │
└─────────────────────────────────────────────┘
```

### 核心优势

1. **简单可靠**
   - 基于标准 HTTP，无需特殊服务器配置
   - 浏览器原生支持，兼容性好
   - 调试简单，问题排查容易

2. **资源高效**
   - 页面不可见时自动停止，节省带宽
   - 无需维持长连接，服务器压力小
   - 支持水平扩展，无需特殊配置

3. **用户体验**
   - 30 秒延迟对法律讨论场景完全可接受
   - 用户操作后立即刷新，体感实时
   - 降级方案：手动刷新按钮

## 💡 实现细节

### 智能轮询 Hook

文件位置：`src/hooks/useDiscussionPolling.ts`

```typescript
// 使用示例
const { triggerUpdate } = useDiscussionPolling(
  async () => {
    await fetchDiscussions();
  },
  {
    interval: 30000,      // 30 秒轮询
    enabled: true,        // 启用轮询
    pollWhenHidden: false // 页面隐藏时停止
  }
);

// 用户发送消息后立即刷新
const handleSend = async () => {
  await sendMessage();
  triggerUpdate(); // 不等待定时器
};
```

### 关键特性

- ✅ 自动管理定时器生命周期
- ✅ 页面可见性检测
- ✅ 防止重复请求
- ✅ 错误处理
- ✅ 内存泄漏防护

## 🔮 未来演进路径

### 何时考虑 WebSocket？

只有在以下条件**同时满足**时才考虑：

1. **业务需求**
   - 用户明确反馈需要实时聊天功能
   - 讨论频率达到 > 10 条/分钟
   - 需要秒级更新的协同编辑功能

2. **技术条件**
   - 产品进入成熟期，有资源投入
   - 用户规模增长，需要更好的体验
   - 服务器支持长连接部署

3. **投入产出**
   - 实时性提升能带来明显的业务价值
   - 技术团队有能力维护 WebSocket 基础设施

### 渐进式升级方案

```
阶段 1 (当前): SSE + 轮询
  ↓ (用户增长，需求明确)
阶段 2: SSE + 通知推送
  ├─ 辩论：SSE 流式推送
  └─ 通知：SSE 推送通知事件
  ↓ (需要实时聊天)
阶段 3: SSE + WebSocket
  ├─ 辩论：SSE 流式推送
  ├─ 通知：WebSocket 双向通信
  └─ 聊天：WebSocket 实时消息
```

## 📈 性能预估

### 当前方案（1000 并发用户）

```
SSE 连接（辩论生成时）:
  - 峰值连接：50-100（假设 5-10% 用户同时生成辩论）
  - 资源占用：低

轮询请求:
  - 每分钟请求：2000（1000 用户 × 2 次/分钟）
  - 单次响应：< 100KB
  - 总带宽：< 200MB/分钟
  - 服务器压力：极低（静态数据缓存）
```

### WebSocket 方案（对比）

```
长连接数：1000（所有在线用户）
资源占用：
  - 内存：1000 × 100KB = 100MB
  - 文件描述符：1000
  - 心跳包：1000 × 60 = 60,000 次/分钟
复杂度：需要连接管理、负载均衡、状态同步
```

**结论**: 当前方案在 MVP 阶段完全够用，且成本更低。
