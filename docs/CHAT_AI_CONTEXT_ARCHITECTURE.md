# 律伴 Chat AI 上下文架构规划

> 形成于 2026-04-03，基于"流程控制 > 提示工程"、Claude Code 源码启示、SCP 语义晶体思想三方向讨论

---

## 一、核心判断

**现状问题**：用户输入原样送给 AI + 通用角色提示 ≈ 用户自己去问 DeepSeek，无差异化。

**解法**：在用户输入和 AI 之间插入受我们控制的**处理流水线**，让 AI 拿到的上下文远比用户的原始文字更结构化、更专业。

---

## 二、目标架构：四阶段流水线

```
用户输入
  │
  ▼
[Stage 1] 案情结构提取          ← 代码逻辑，非 prompt 魔法
  │  提取: 当事人/争议/事实/诉求
  ▼
[Stage 2] 知识检索注入          ← 律伴独有数据资产
  │  查: 指导案例库 / 法条库 / SCP晶体库
  ▼
[Stage 3] 上下文组装            ← prompt engineering 真正发挥的地方
  │  组装: 系统指令 + 案情晶体 + 相关法条 + 最近N轮对话
  ▼
[Stage 4] AI 生成（DeepSeek/GLM）
  │
  ▼
[Stage 5] 结果解析 + 晶体更新   ← 写回 DB，下轮使用
```

---

## 三、SCP 在对话中的角色：案情语义晶体

### 核心思想

不把原始消息历史原样堆给 AI（token 浪费，注意力稀释），而是维护一个随对话演进的**结构化案情摘要**，作为每轮 AI 调用的稳定骨架。

### 案情晶体结构（JSON，存于 conversations.caseContext）

```json
{
  "version": 5,
  "updatedAt": "2026-04-03T10:30:00Z",
  "case_type": "劳动争议",
  "parties": {
    "plaintiff": "张某，劳动者，工龄12年",
    "defendant": "某科技公司"
  },
  "core_dispute": "劳动合同解除是否合法",
  "established_facts": [
    { "fact": "2024年3月解除劳动关系", "confidence": "CONFIRMED" },
    { "fact": "解除时无书面通知", "confidence": "CONFIRMED" },
    { "fact": "存在违纪记录", "confidence": "UNCERTAIN" }
  ],
  "applicable_law": [
    { "article": "劳动合同法第48条", "status": "CONFIRMED_APPLICABLE" },
    { "article": "第39条", "status": "PENDING_VERIFICATION" }
  ],
  "current_position": "原告主张违法解除，要求赔偿金",
  "open_questions": ["违纪记录是否经过工会程序？", "是否签署竞业限制协议？"],
  "documents_uploaded": []
}
```

### 晶体替代历史消息的策略

| 对话阶段   | 送给 AI 的上下文                             |
| ---------- | -------------------------------------------- |
| 第 1 轮    | system + 用户原文                            |
| 第 2 轮起  | system + **案情晶体** + 最近 6 轮对话        |
| 超过 20 轮 | system + 案情晶体 + 最近 6 轮 + **中期摘要** |

无论对话多长，有效 token 密度始终保持高水位。

## 四、Claude Code 源码启示与对应措施

| Claude Code 技术            | 律伴对应实施                                                     |
| --------------------------- | ---------------------------------------------------------------- |
| Tool Use（AI 主动调用工具） | 未来：给 AI 提供 `search_law()`、`get_case()` 工具，让它自主检索 |
| 对话压缩（Compaction）      | 中期：超长对话自动压缩为中期摘要，注入下轮                       |
| 外部状态（CLAUDE.md）       | **现在**：案情晶体存 DB，每轮注入                                |
| Agent Loop（多步骤）        | 未来：复杂问题拆解为子任务链，逐步执行                           |

## 五、实施路线图

### Tier 1 — 立即实施（本次）

- [x] **改进 system prompt**：加入阶段感知指令、输出规范、晶体利用声明
- [x] **Schema**：`conversations` 表新增 `caseContext Json?` 字段
- [x] **晶体提取**：第一轮对话完成后，异步调用 AI 提取案情晶体写回 DB
- [x] **晶体注入**：第二轮起，在 AI 上下文中前置案情晶体

### Tier 2 — 本周

- [ ] **法条检索注入**：用户消息触发关键词提取 → 检索 law_articles 表 → 相关条文注入上下文（最多 3 条）
- [ ] **指导案例注入**：同上，检索 guiding_cases 表，把最相关案例摘要注入
- [ ] **对话标题自动优化**：晶体提取后用 `core_dispute` 字段更新对话标题

### Tier 3 — 下周

- [ ] **AI Tool Use 接入**：定义 `search_law_article`、`retrieve_guiding_case` 工具，让 AI 在生成过程中主动调用
- [ ] **对话压缩**：当 history 超过 20 条时，把 6 条以前的历史压缩为中期摘要存入晶体

### Tier 4 — 中期

- [ ] **用户画像上下文**：注入用户角色（律师/法务）、历史案件偏好、当前关联案件
- [ ] **跨对话知识复用**：同一用户在不同对话里确认过的法律观点，可跨会话引用
- [ ] **晶体可视化**：在 UI 侧边栏展示当前案情晶体，用户可编辑

---

## 六、关键设计约束

1. **晶体提取不阻塞用户响应**：异步提取，不在 POST /messages 的关键路径上
2. **晶体置信度继承 SCP 语义**：CONFIRMED / UNCERTAIN / REFUTED 与现有 epistemic state 一致
3. **晶体不替代原文**：原始消息永远保留在 DB，晶体只是压缩索引
4. **降级安全**：晶体提取失败时，回退到原始历史消息方式，不影响基本功能
